01FUNDAMENTO

Memoria y persistencia

Punto de partida

La clase pasada creaste productos dentro de un arreglo. Si el servidor se apaga, ¿qué ocurre con los datos de una tienda real?

En la clase anterior, los productos vivían dentro de un arreglo de JavaScript. Mientras NestJS estaba encendido, todo parecía funcionar: podías crear un producto, consultarlo y verlo en la respuesta.

El problema aparece cuando el proceso termina. El arreglo estaba en la memoria RAM del servidor; al reiniciar NestJS, esa memoria se libera y el archivo vuelve a ejecutarse desde cero.

Modelo mental

Cómo imaginarlo

La memoria RAM funciona como un tablero de clase: permite escribir rápido, pero lo que queda allí desaparece cuando se borra. PostgreSQL funciona como un cuaderno: tarda un poco más en guardar, pero conserva la información aunque el servidor se apague.

Definición técnica

La persistencia es la capacidad de conservar información más allá de la ejecución actual de un programa.

En palabras sencillas

Los datos dejan de depender de que NestJS esté encendido.

Ejemplo real

Un pedido creado hoy debe seguir existiendo mañana, incluso si el servidor se reinicia durante la noche.

MODELO Flujo de la clase anterior 
Petición POST → Controlador → Servicio → Array en RAM → Respuesta
MODELO Flujo con persistencia 
Petición HTTP → Controlador → Servicio → Prisma ORM → PostgreSQL → Disco

Antes de continuar

¿El único problema de un arreglo es que se borra al reiniciar?
No. También aparecen problemas de integridad, concurrencia, relaciones, consultas y respaldo. Persistir es solo el primer paso para tratar los datos como un recurso real del negocio.
ProblemaEjemplo en la tienda
PersistenciaEl pedido debe existir mañana.
IntegridadUn producto no puede pertenecer a una categoría inexistente.
UnicidadNo puede haber dos usuarios con el mismo correo.
ConcurrenciaVarias personas pueden comprar al mismo tiempo.
ConsultasBuscar productos por precio, categoría o stock.
02FUNDAMENTO

SQL y NoSQL

Guardar datos en un archivo también permite sobrevivir a un reinicio, pero un archivo no está diseñado para resolver búsquedas complejas, acceso simultáneo, restricciones ni relaciones.

Base de datos

Sistema especializado en almacenar, organizar y recuperar información de forma segura, permanente y eficiente.

SQL

Organiza los datos en tablas relacionadas, con una estructura definida de antemano.

NoSQL

Usa estructuras más flexibles, como documentos parecidos a JSON, que no siempre comparten exactamente los mismos campos.

Usa SQL cuando...Usa NoSQL cuando...
Los datos tienen estructura estable.La estructura cambia con frecuencia.
Hay relaciones fuertes entre entidades.Los datos son mayormente independientes.
La consistencia es crítica.La flexibilidad y la escala pesan más.
Ejemplo: ventas, inventario, nómina.Ejemplo: logs, caché, contenido flexible.

Caso de negocio

Una tienda con relaciones

Un usuario realiza pedidos, una categoría agrupa productos y cada pedido contiene varios productos. Este dominio necesita relaciones claras y consistencia, por eso PostgreSQL es una elección adecuada.

Antes de continuar

¿Cuál es mejor: SQL o NoSQL?
Ninguna es mejor en todos los casos. La pregunta profesional es cuál resuelve mejor el problema actual. En esta tienda necesitamos relaciones y consistencia, por eso usaremos PostgreSQL.
03MODELADO

Entidades y relaciones

Antes de crear tablas, debes entender qué conceptos existen en el negocio y cómo se conectan. Modelar primero evita construir estructuras difíciles de mantener.

Cómo detectar entidades

Escucha los sustantivos del dominio: los usuarios hacen pedidos de productos que pertenecen a categorías. Cada concepto con identidad propia puede convertirse en una entidad.

Entidad

Concepto del negocio del que necesitamos guardar información.

Clave primaria

Identificador único de cada fila. Funciona como una cédula.

Clave foránea

Columna que guarda la clave primaria de otra tabla y crea la relación.

Relación uno a muchos

Una categoría puede tener muchos productos, pero cada producto pertenece a una sola categoría. La clave foránea se guarda en el lado “muchos”: Producto guarda categoriaId.

Relación muchos a muchos

Un pedido puede contener muchos productos y un producto puede aparecer en muchos pedidos. Esta relación necesita una tabla intermedia: DetallePedido.

Decisión de modelado

El precio histórico

El detalle del pedido debe guardar la cantidad y el precio pagado en ese momento. Si el precio del producto cambia mañana, las facturas antiguas no deben modificarse.

MODELO Diagrama del dominio 
Usuario 1 ─── N Pedido
Categoria 1 ─── N Producto
Pedido 1 ─── N DetallePedido
Producto 1 ─── N DetallePedido

Antes de continuar

¿Por qué no guardar todo en una sola tabla gigante?
Porque repetirías datos, crearías contradicciones y harías más costosos los cambios. Separar entidades reduce redundancia y permite que cada dato tenga un lugar claro.
04FUNDAMENTO

ORM y Prisma

PostgreSQL trabaja con tablas y SQL. NestJS trabaja con objetos y TypeScript. Un ORM conecta esos dos mundos.

Definición técnica

ORM significa Object-Relational Mapping. Traduce operaciones sobre objetos a consultas SQL.

En palabras sencillas

Tú escribes TypeScript y Prisma genera el SQL necesario.

Ejemplo real

Funciona como un traductor: tu aplicación habla TypeScript y PostgreSQL habla SQL.

COMPARAR Consulta escrita en SQL 
SELECT id, nombre, precio FROM productos WHERE stock > 0;
COMPARAR Consulta escrita con Prisma 
return this.prisma.producto.findMany({
  where: { stock: { gt: 0 } },
});

Buena práctica

El ORM no reemplaza el conocimiento de SQL. Debes saber leer las consultas y entender relaciones para detectar problemas de rendimiento o modelado.

Error frecuente

Qué ocurre: la conexión falla aunque el usuario y la contraseña parezcan correctos.

Cómo leerlo: revisa si la contraseña contiene caracteres como @, # o /.

Cómo resolverlo: codifica esos caracteres en formato URL, por ejemplo @ como %40.

export default defineConfig({

  migrations: {

06IMPLEMENTACIÓN

Crear el modelo y migrar

Ahora convertiremos el modelo del negocio en un schema que Prisma pueda interpretar.

Qué vamos a hacer

Definiremos usuarios, categorías, productos, pedidos y detalles.

Por qué lo hacemos

El schema debe representar las reglas del negocio antes de crear tablas reales.

Qué aprenderás

Cómo se expresan claves, restricciones, relaciones y tipos de datos en Prisma.

AGREGAR prisma/schema.prisma · modelos 
enum EstadoPedido {
  PENDIENTE
  PAGADO
  ENVIADO
  ENTREGADO
  CANCELADO
}

model Usuario {
  id       Int      @id @default(autoincrement())
  nombre   String
  email    String   @unique
  creadoEn DateTime @default(now())
  pedidos  Pedido[]

  @@map("usuarios")
}

model Categoria {
  id        Int        @id @default(autoincrement())
  nombre    String     @unique
  creadoEn  DateTime   @default(now())
  productos Producto[]

  @@map("categorias")
}

model Producto {
  id            Int             @id @default(autoincrement())
  nombre        String
  descripcion   String?
  precio        Decimal         @db.Decimal(12, 2)
  stock         Int             @default(0)
  categoriaId   Int
  creadoEn      DateTime        @default(now())
  actualizadoEn DateTime        @updatedAt
  categoria     Categoria       @relation(fields: [categoriaId], references: [id])
  detalles      DetallePedido[]

  @@index([categoriaId])
  @@map("productos")
}

model Pedido {
  id        Int             @id @default(autoincrement())
  estado    EstadoPedido    @default(PENDIENTE)
  total     Decimal         @db.Decimal(12, 2)
  usuarioId Int
  creadoEn  DateTime        @default(now())
  usuario   Usuario         @relation(fields: [usuarioId], references: [id])
  detalles  DetallePedido[]

  @@index([usuarioId])
  @@map("pedidos")
}

model DetallePedido {
  id             Int      @id @default(autoincrement())
  cantidad       Int
  precioUnitario Decimal  @db.Decimal(12, 2)
  pedidoId       Int
  productoId     Int
  pedido         Pedido   @relation(fields: [pedidoId], references: [id], onDelete: Cascade)
  producto       Producto @relation(fields: [productoId], references: [id])

  @@unique([pedidoId, productoId])
  @@index([productoId])
  @@map("detalles_pedido")
}

Decisión importante

Para dinero se utiliza Decimal, no Float. Los valores monetarios necesitan precisión exacta y no deben acumular errores de redondeo.

Aplicar la primera migración

Qué vamos a hacer

Por qué lo hacemos

El archivo schema.prisma describe el estado deseado, pero PostgreSQL todavía no conoce esas tablas.

Qué aprenderás

07ARQUITECTURA

Conectar Prisma con NestJS

EJECUTAR Terminal · generar infraestructura Prisma 
nest generate module prisma
nest generate service prisma --no-spec

Qué vamos a hacer

Crearemos un servicio inyectable que hereda las operaciones de PrismaClient.

Por qué lo hacemos

La aplicación necesita abrir la conexión al iniciar y cerrarla al apagarse.

Qué aprenderás

Cómo integrar Prisma con el ciclo de vida y la inyección de dependencias de NestJS.

REEMPLAZAR src/prisma/prisma.service.ts 
import { Injectable, OnModuleDestroy, OnModuleInit } from '@nestjs/common';
import { PrismaPg } from '@prisma/adapter-pg';
import { PrismaClient } from '../generated/prisma/client';

@Injectable()
export class PrismaService
  extends PrismaClient
  implements OnModuleInit, OnModuleDestroy
{
  constructor() {
    const connectionString = process.env.DATABASE_URL;

    if (!connectionString) {
      throw new Error('DATABASE_URL no está definida. Revisa el archivo .env.');
    }

    const adapter = new PrismaPg({ connectionString });
    super({ adapter });
  }

  async onModuleInit(): Promise<void> {
    await this.$connect();
  }

  async onModuleDestroy(): Promise<void> {
    await this.$disconnect();
  }
}

Qué vamos a hacer

Exportaremos PrismaService desde un módulo global.

Por qué lo hacemos

Productos, categorías, usuarios y pedidos necesitarán la misma conexión.

Qué aprenderás

Cómo compartir infraestructura transversal sin repetir imports en cada módulo.

REEMPLAZAR src/prisma/prisma.module.ts 
import { Global, Module } from '@nestjs/common';
import { PrismaService } from './prisma.service';

@Global()
@Module({
  providers: [PrismaService],
  exports: [PrismaService],
})
export class PrismaModule {}

Categorías

nest generate controller categorias --no-spec
nest generate service categorias --no-spec

Qué vamos a hacer

Implementaremos creación y consulta de categorías usando Prisma.

Por qué lo hacemos

Un producto necesita una categoría existente para mantener la relación.

Qué aprenderás

Cómo usar create, findMany, include y orderBy.

REEMPLAZAR src/categorias/categorias.service.ts 
import { Injectable } from '@nestjs/common';
import { PrismaService } from '../prisma/prisma.service';

type CrearCategoriaInput = {
  nombre: string;
};

@Injectable()
export class CategoriasService {
  constructor(private readonly prisma: PrismaService) {}

  crear(datos: CrearCategoriaInput) {
    return this.prisma.categoria.create({
      data: { nombre: datos.nombre },
    });
  }

  obtenerTodas() {
    return this.prisma.categoria.findMany({
      include: { productos: true },
      orderBy: { id: 'asc' },
    });
  }
}
REEMPLAZAR src/categorias/categorias.controller.ts 
import { Body, Controller, Get, Post } from '@nestjs/common';
import { CategoriasService } from './categorias.service';

type CrearCategoriaBody = {
  nombre: string;
};

@Controller('categorias')
export class CategoriasController {
  constructor(private readonly categoriasService: CategoriasService) {}

  @Post()
  crear(@Body() body: CrearCategoriaBody) {
    return this.categoriasService.crear(body);
  }

  @Get()
  obtenerTodas() {
    return this.categoriasService.obtenerTodas();
  }
}

Productos

Qué vamos a hacer

Eliminaremos el arreglo y consultaremos directamente PostgreSQL.

Por qué lo hacemos

La base debe asignar los identificadores y conservar los registros.

Qué aprenderás

Cómo crear, listar y buscar productos incluyendo su categoría.

REEMPLAZAR src/productos/productos.service.ts 
import { Injectable } from '@nestjs/common';
import { PrismaService } from '../prisma/prisma.service';

type CrearProductoInput = {
  nombre: string;
  descripcion?: string;
  precio: number;
  stock: number;
  categoriaId: number;
};

@Injectable()
export class ProductosService {
  constructor(private readonly prisma: PrismaService) {}

  crear(datos: CrearProductoInput) {
    return this.prisma.producto.create({
      data: datos,
      include: { categoria: true },
    });
  }

  obtenerTodos() {
    return this.prisma.producto.findMany({
      include: { categoria: true },
      orderBy: { id: 'asc' },
    });
  }

  obtenerPorId(id: number) {
    return this.prisma.producto.findUnique({
      where: { id },
      include: { categoria: true },
    });
  }
}
REEMPLAZAR src/productos/productos.controller.ts 
import { Body, Controller, Get, Param, ParseIntPipe, Post } from '@nestjs/common';
import { ProductosService } from './productos.service';

type CrearProductoBody = {
  nombre: string;
  descripcion?: string;
  precio: number;
  stock: number;
  categoriaId: number;
};

@Controller('productos')
export class ProductosController {
  constructor(private readonly productosService: ProductosService) {}

  @Post()
  crear(@Body() body: CrearProductoBody) {
    return this.productosService.crear(body);
  }

  @Get()
  obtenerTodos() {
    return this.productosService.obtenerTodos();
  }

  @Get(':id')
  obtenerPorId(@Param('id', ParseIntPipe) id: number) {
    return this.productosService.obtenerPorId(id);
  }
}

Arquitectura en acción

El controlador cambia muy poco. La fuente de datos cambió del arreglo a PostgreSQL dentro del servicio. Esta separación evita que la capa HTTP dependa de la tecnología de persistencia.

Prueba de fuego

Apaga y vuelve a encender

Consulta los productos, detén NestJS con Ctrl + C, vuelve a iniciar con npm run start:dev y repite la consulta. El producto debe seguir allí.

Antes de continuar

¿Por qué el producto sigue existiendo después del reinicio?
Porque ya no vive dentro del proceso de NestJS. Ahora está almacenado en PostgreSQL, en disco. El servidor puede reiniciarse sin borrar los datos.

Errores frecuentes

P1012: revisa que la URL no esté dentro de schema.prisma.

Authentication failed: verifica usuario, contraseña y caracteres especiales.

Can't reach database server: confirma que PostgreSQL esté activo y use el puerto correcto.

Cannot find module: ejecuta npx prisma generate y revisa la ruta de salida.
