01 · EL GRAN SALTO

🚀 Del CLI al backend

En el módulo anterior construiste programas que corrían en tu computador, en tu terminal, pedían datos por teclado y, al terminar, se apagaban. Perfecto para aprender lógica. Pero imagina que una empresa quiere que 500 personas usen tu programa al mismo tiempo, cada una desde su propio computador. Ese programa ya no sirve.

Lo que hace falta es un programa distinto: uno que viva en un computador central, esté siempre encendido y atienda a mucha gente a la vez. Eso es un backend, y hoy construyes el tuyo.

Backend: el programa que vive en un servidor (un computador central siempre encendido) y responde a las peticiones de otros programas o usuarios.

Es como la cocina de un restaurante: tú no la ves desde la mesa, pero está ahí, encendida, atendiendo todos los pedidos que llegan de todas las mesas al tiempo.

Qué cambia respecto al módulo anterior

Antes: un usuario (tú) · el programa arrancaba y terminaba · pedías datos por teclado · corrías un archivo con node.

Ahora: muchos usuarios a la vez · el servidor queda encendido esperando · los datos llegan por peticiones a URLs · un framework (NestJS) te da la estructura.

Si le pasaras tu programa del módulo anterior a 500 personas por WhatsApp, ¿todas verían y compartirían la misma información?
▸ Ver explicación
No. Cada persona correría su propia copia aislada, con sus propios datos, sin nada en común. Por eso necesitas un solo programa central al que todos se conecten: un backend. Es justo lo que hoy vas a levantar.

Al final de esta clase vas a tener una API de un catálogo de productos: cuando alguien la consulte, devuelve la lista; cuando alguien envíe un producto nuevo, lo agrega.

API: la forma en que dos programas se hablan entre sí, con reglas claras de qué pedir y qué se responde.

Es el mesero de un restaurante: tú no entras a la cocina, le pides al mesero y él te trae. Hoy tú construyes ese mesero para que otros programas le hagan pedidos.

02 · LA HERRAMIENTA

🧩 Qué es NestJS

NestJS es un framework de Node.js para construir el backend de una aplicación. Está escrito en TypeScript y organiza el código en tres piezas que se repiten siempre: módulos, controladores y servicios.

Es un restaurante bien montado: cada persona tiene un solo trabajo y todo el mundo sabe dónde está cada cosa.

Con Node.js a secas ya se pueden hacer servidores. Entonces, ¿por qué las empresas le ponen encima un framework que las obliga a organizar el código de cierta forma? Porque cuando un proyecto crece y lo tocan muchas personas durante años, la libertad total se vuelve caos: cada quien ordena distinto y nadie encuentra nada.

NestJS impone una estructura común y predecible. Un desarrollador nuevo entra a cualquier proyecto Nest y sabe de inmediato dónde está todo. La regla no es una jaula: es un idioma compartido.

Idea clave

Un framework no te quita libertad porque sí: te la cambia por orden. Y en equipos, el orden vale más que la libertad.

¿Por qué sería un problema que cada desarrollador de un equipo organizara el código a su manera?
▸ Ver explicación
Porque nadie entendería el trabajo de los demás sin explicaciones largas, y cada cambio de persona costaría semanas. Una estructura común hace que el código sea de todos, no solo de quien lo escribió. Ese es el mayor regalo de NestJS.
03 · CONCEPTOS NUEVOS

🏷️ Decoradores e inyección de dependencias

Antes de escribir código, dos ideas nuevas que verás por todos lados en Nest. Si estas quedan claras, el resto se vuelve fácil.

Los decoradores (ese símbolo @)

Un decorador es una etiqueta que le pega información extra a una clase, un método o un parámetro, para que Nest sepa qué papel cumple. No cambia lo que hace por dentro; le añade un rol.

Es el gafete de un empleado: la persona es la misma, pero el gafete dice "MESERO" o "COCINERO". Nest lee el gafete y sabe qué hacer con cada clase.

Por ejemplo, @Controller('productos') es un gafete que dice: "esta clase es el mesero de la sección productos".

La inyección de dependencias

Inyección de dependencias: cuando una clase recibe ya listas las herramientas que necesita, en lugar de fabricarlas ella misma. Nest las crea y se las entrega.

El mesero no fabrica a su cocinero. Al llegar a trabajar, el restaurante ya le asignó uno y se lo puso al lado, listo para usar. El mesero solo lo pide.

Por qué importa

El mesero no tiene que saber cómo se construye un cocinero: solo lo usa. Eso hace el código más simple y más fácil de mantener. Lo verás en acción cuando escribamos el controlador.

Cuando escribas this.productosService.obtenerTodos() sin haber creado el servicio con new, ¿de dónde salió ese servicio?
▸ Ver explicación
Salió del constructor: ahí Nest inyectó el servicio ya construido y te lo entregó listo. Tú solo lo declaraste como algo que necesitas; el framework se encargó de dártelo. Eso es la inyección de dependencias.
04 · EL MODELO MENTAL

🍴 Mesero, cocinero y restaurante

Toda la clase gira alrededor de tres archivos que trabajan en equipo. Guárdalos bien, porque los verás en cada proyecto Nest de aquí en adelante.

🧑‍💼 Controlador (*.controller.ts) — el mesero. Recibe la petición y devuelve la respuesta. No cocina.

👨‍🍳 Servicio (*.service.ts) — el cocinero. Hace la lógica y maneja los datos. No habla con el cliente.

🏠 Módulo (*.module.ts) — el restaurante. Agrupa y conecta al mesero con su cocinero.

El flujo, paso a paso

Recórrelo mentalmente

Llega una petición → el mesero la recibe → le pide al cocinero que haga el trabajo → el cocinero devuelve el resultado → el mesero lo responde a quien preguntó.

¿Por qué separar mesero y cocinero? ¿No sería más fácil que uno solo hiciera todo?
▸ Ver explicación
Porque mezclar responsabilidades vuelve el código imposible de mantener. Si el mesero cocina y atiende, cuando algo falle no sabrás si el problema fue atendiendo o cocinando. Separando, cada pieza se prueba y se corrige por aparte. Es la misma separación de responsabilidades que ya aplicaste al partir un proyecto en varios archivos.
05 · MANOS A LA OBRA

⚙️ Instalar y crear el proyecto

1. Verifica tu versión de Node.js

NestJS 11 necesita Node.js 20 o superior. En la terminal (en VS Code: menú Terminal → New Terminal) escribe:

terminal 
node --version

Debe aparecer v20.x.x, v22.x.x o superior. Si sale una versión menor, entra a nodejs.org, descarga la versión LTS (la recomendada), instálala como cualquier programa, reinicia la terminal y vuelve a verificar.

2. Instala el CLI de NestJS

El CLI es una herramienta que crea proyectos y archivos por ti, ya organizados. Se instala una sola vez en tu computador (la bandera -g significa "global", disponible para todos tus proyectos):

terminal 
npm install -g @nestjs/cli

Confirma que quedó instalado:

terminal 
nest --version

3. Crea el proyecto

Primero, ubica la terminal en la carpeta donde guardas tus proyectos (con cd, "change directory"). Por ejemplo:

terminal 
cd Documentos/dev-senior

Y ahí, crea el proyecto:

terminal 
nest new catalogo-api

El nombre catalogo-api es solo el que elegimos: podría ser otro, pero siempre en minúsculas, sin espacios (usa guiones) y sin tildes ni ñ. Cuando el CLI pregunte el gestor de paquetes, elige npm con las flechas y Enter. La instalación tarda uno o dos minutos.

¿Y si hago clic derecho → New Folder?

nest new no crea una carpeta vacía: crea una casa ya amoblada con toda la configuración y las dependencias instaladas, lista para arrancar. Una carpeta vacía sería un lote de tierra: te tocaría construir todo a mano.

06 · ENCENDER

🖥️ Encender el servidor

Entra a la carpeta que se creó y arranca el servidor en modo desarrollo:

terminal 
cd catalogo-api
npm run start:dev

Aparecen varias líneas verdes. La clave es la última: Nest application successfully started. Si sale eso, tu servidor está encendido y escuchando.

start:dev

Este modo vigila tus cambios: cada vez que guardas un archivo, el servidor se reinicia solo para aplicarlo. Comodísimo mientras desarrollas.

Míralo responder

Un servidor no se prueba en la terminal, se prueba en el navegador. Abre Chrome y entra a:

navegador 
http://localhost:3000

Debe aparecer el texto Hello World!. Ese texto lo generó y lo envió tu propio servidor. Acabas de correr tu primer backend. 🎉

localhost significa "este mismo computador". El 3000 es el puerto: la puerta numerada por donde tu servidor escucha.

Es como una dirección: localhost es el edificio (tu computador) y 3000 es el número del apartamento donde vive tu servidor. Para visitarlo necesitas los dos datos.

En el módulo anterior tus programas terminaban solos. ¿Por qué este se queda "encendido" sin cerrarse?
▸ Ver explicación
Porque un servidor no hace una tarea y se va: se queda escuchando peticiones indefinidamente, listo para responder cuando alguien llegue. Esa es la diferencia esencial entre un programa de una sola corrida y un backend.
07 · ORIENTARSE

📂 Los archivos base

El proyecto trae muchos archivos, pero todo tu código vive en la carpeta src/. El resto es configuración que hoy no tocas. Dentro de src/, los que importan:

main.ts — el botón de encendido. El primero que se ejecuta.

app.module.ts — el restaurante principal.

app.controller.ts — un mesero de ejemplo (el del "Hello World").

app.service.ts — un cocinero de ejemplo.

main.ts — el encendido

Este archivo crea la aplicación y le dice "ponte a escuchar en el puerto 3000". Casi nunca lo modificas, pero es bueno saber qué hace.

src/main.ts (solo lo leemos) 
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();

La línea clave es app.listen(... 3000): es la que pone al servidor a escuchar y hace que no se apague.

app.module.ts — el restaurante

El decorador @Module declara "esto es un restaurante" y adentro registra sus meseros (controllers) y cocineros (providers).

src/app.module.ts (solo lo leemos) 
import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';

@Module({
  imports: [],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
El corazón de todo módulo

Esas tres listas — imports, controllers, providers — aparecen en cada módulo de Nest. Vale la pena reconocerlas de memoria.

08 · CONSTRUIR

🛒 Tu API de productos

Dejamos el "Hello World" y construimos algo propio: una sección de productos con su mesero, su cocinero y su registro en el restaurante. En lugar de crear los tres archivos a mano, el CLI los genera ya conectados.

1. Genera el módulo

Con la terminal ubicada dentro de la carpeta del proyecto (catalogo-api):

terminal · dentro de catalogo-api 
nest generate module productos

Verás CREATE del archivo nuevo y, sobre todo, UPDATE src/app.module.ts: Nest abrió tu archivo principal y registró solo el nuevo módulo en la lista imports. Eso es el framework encargándose de las conexiones por ti.

2. Genera el controlador y el servicio

terminal · dentro de catalogo-api 
nest generate controller productos
nest generate service productos

También se crean archivos que terminan en .spec.ts: son para pruebas automáticas. Por ahora los ignoramos y no los borramos.

3. El cocinero: escribe el servicio

El servicio guarda los datos y la lógica. Por ahora los productos viven en un array en memoria (sin base de datos todavía). Es el mismo tipo de array de objetos que ya usaste antes; lo nuevo es el @Injectable() de arriba.

src/productos/productos.service.ts — reemplaza todo 
import { Injectable } from '@nestjs/common';

@Injectable()
export class ProductosService {
  private productos = [
    { id: 1, nombre: 'Teclado mecánico', precio: 120000, stock: 15 },
    { id: 2, nombre: 'Mouse inalámbrico', precio: 45000, stock: 30 },
  ];

  obtenerTodos() {
    return this.productos;
  }

  crear(nuevo: { nombre: string; precio: number; stock: number }) {
    const producto = {
      id: this.productos.length + 1,
      nombre: nuevo.nombre,
      precio: nuevo.precio,
      stock: nuevo.stock,
    };
    this.productos.push(producto);
    return producto;
  }
}

@Injectable() → el gafete que dice "soy un cocinero; Nest puede inyectarme donde me pidan".

private productos → la despensa: solo el cocinero la toca desde adentro.

obtenerTodos() → devuelve la lista completa.

crear(nuevo) → arma un producto con un id nuevo, lo agrega con .push() y lo devuelve.

4. El mesero: escribe el controlador

El controlador recibe las peticiones y le pide al servicio que trabaje. Aquí aparecen @Get() (consultar), @Post() (crear) y @Body() (leer los datos que llegan).

src/productos/productos.controller.ts — reemplaza todo 
import { Body, Controller, Get, Post } from '@nestjs/common';
import { ProductosService } from './productos.service';

@Controller('productos')
export class ProductosController {
  constructor(private readonly productosService: ProductosService) {}

  @Get()
  obtenerTodos() {
    return this.productosService.obtenerTodos();
  }

  @Post()
  crear(@Body() nuevo: { nombre: string; precio: number; stock: number }) {
    return this.productosService.crear(nuevo);
  }
}

@Controller('productos') → el gafete del mesero: todo lo que llegue a /productos es suyo.

constructor(...productosService...) → aquí Nest inyecta el cocinero y se lo pone al lado al mesero.

@Get() → responde cuando alguien consulta /productos.

@Post() → responde cuando alguien envía un producto nuevo a /productos.

@Body() → lee los datos que vienen en la petición (como leer el papelito del pedido).

GET y POST son métodos HTTP: los verbos que dicen qué quiere hacer quien te habla. GET = leer/consultar (no cambia nada). POST = crear/enviar algo nuevo.

GET es "muéstrame la carta"; POST es "anota este nuevo pedido". Hay más (PUT para actualizar, DELETE para borrar) y los verás en la próxima clase.

09 · PROBAR

🔎 Probar la API

Guarda todo. Como el servidor corre con start:dev, se reinicia solo. En la terminal del servidor aparecen las rutas mapeadas: Mapped {/productos, GET} y Mapped {/productos, POST}.

Consultar (GET)

En el navegador entra a:

navegador 
http://localhost:3000/productos

Aparece la lista con los dos productos de ejemplo, en formato JSON. 🎉

JSON es el formato estándar en que las APIs mandan datos. Se parece muchísimo a los objetos y arrays de JavaScript que ya conoces.

Es el idioma común en que los programas se pasan información, sin importar en qué lenguaje esté escrito cada uno.

Crear (POST)

El navegador, escribiendo una URL, solo sabe hacer GET. Para crear un producto se hace una petición POST enviando los datos. Así se ve desde la terminal:

terminal (nueva) 
curl -X POST http://localhost:3000/productos -H "Content-Type: application/json" -d "{\"nombre\":\"Monitor 24\",\"precio\":650000,\"stock\":8}"

Responde con el producto creado, ya con su id: 3. Si recargas http://localhost:3000/productos en el navegador, el "Monitor 24" ya aparece en la lista.

Próxima clase

Probar un POST así, escribiendo comandos largos, es incómodo. Por eso la próxima clase instalamos Postman, una herramienta hecha justo para probar APIs con comodidad.

10 · GUARDAR

☁️ Subir a GitHub

Como siempre, el trabajo se sube. Buena noticia: al crear el proyecto, Nest ya inicializó el repositorio Git y dejó un .gitignore que evita subir node_modules. Lo que falta es tu primer commit:

terminal · dentro de catalogo-api 
git add .
git commit -m "Clase 1 Modulo 2: setup NestJS y primera API de productos"

Crear el repositorio en GitHub y hacer git push ya lo dominas del módulo anterior.

¿Por qué es buena idea que node_modules nunca se suba a GitHub?
▸ Ver explicación
Porque son miles de archivos pesados que cualquiera puede regenerar en segundos con npm install, leyendo la lista de dependencias del package.json. Subirlos solo ensucia y engorda el repositorio. Por eso el .gitignore los excluye a propósito.
